# Design canon

Normative decisions for the project — the constraints the code must satisfy, each an
addressable `D-N` unit that tickets and agents cite as `per D-N`. Decisions and
constraints only: never a description of how the code works, and never a plain
canonical name (that goes in `docs/vocabulary.md`).

Writing canon — what earns a `D-N`, the entry format, the `Why:` that bounds a
decision's scope — is the `domain-modeling` skill's job.
